# クラウドサイン署名フロー設計

## 想定フロー概要

- 契約書はクラウドサインAPIから送信する。
- 署名フローは以下の順番で進むことを想定する。
  1. 法務担当者
  2. 自社の承認者（最終決裁者）
  3. 相手側担当者

※ 3者ともクラウドサイン上は「署名者」として扱われる想定（3者締結）。

## APIレベルでの前提

- Web API はクラウドサインの公式仕様に従う。
- 主なエンドポイント:
  - `POST /documents` : 書類（下書き）の作成
  - `POST /documents/{documentID}/files` : 契約書PDFの添付
  - `POST /documents/{documentID}/participants` : 署名者（participants）の登録
  - `POST /documents/{documentID}` : 書類の送信

## participants の設計イメージ

`POST /documents/{documentID}/participants` に、以下の順序で participants を登録する。

1. 法務担当者
   - 役割: 契約内容チェック兼署名者
2. 自社承認者
   - 役割: 最終承認兼署名者
3. 相手側担当者
   - 役割: 取引先としての署名者

この順番でメールが送られ、クラウドサイン上の署名処理も同じ順序で行われる想定とする。

## 社内承認との切り分け

- 「サインせずに承認だけする社内フロー」は、クラウドサインAPIの participants だけでは明示的に表現しにくいため、
  - 法務レビューのみでサインさせたくない場合は、別途社内ワークフロー（Notion / スプレッドシート / ワークフローSaaS など）で完結させる。
  - 承認完了後に、初めてクラウドサインAPIで `/documents` を作成・送信する。

- 本ドキュメントでは、「法務も含めて3者がクラウドサイン上で署名する」3者締結パターンを前提にする。
