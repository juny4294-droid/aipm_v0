# クラウドサイン「承認機能」をAPIで使う方法 — 調査メモ

## 結論（要約）

- **order は「宛先の順序」だけ。** order（＝participants の追加順）を設定しても、**それだけでは押印者にはならない。** クラウドサインの「確認機能」は**押印者とは別の扱い**である。
- **押印者 vs 確認者**は、**その宛先に入力項目（押印・フリーテキスト等）を割り当てるかどうか**で決まる：
  - **割り当てる** → その宛先は押印・入力の上で同意 ＝ 押印者（締結者）
  - **割り当てない** → その宛先は「同意」ボタンだけで完了 ＝ 実質確認者
- API では **`POST .../files/{fileID}/widgets`** で入力項目を追加する際に **`participant_id`** を指定する。確認者にしたい宛先には **widget を一切追加しない**ことで、「承認のみ」のフローを表現できる。
- 宛先の**順序**は、`POST /documents/{documentID}/participants` で追加する**順番**で表現可能（追加順にメールが送られ、処理される）。

---

## 1. UI上の「承認」まわり

- 宛先を複数設定すると、**設定した順番に**確認依頼メールが送られる（順次送付）。
- 宛先には例えば「自社の上長」「取引先担当者」などを登録でき、**上長承認 → 先方送付**のような運用が可能。
- ヘルプ上の用語:
  - **締結者**: 契約に同意し、電子署名を行う者
  - **確認者**: 書類に不備がないか確認する者（署名は行わない運用も想定）
- 同一メールアドレスは原則1回のみ（送信者を除く）。宛先上限は127件。

参照: クラウドサインヘルプ「宛先を複数設定して、書類の送付順を設定する」等。

## 2. Web API の participants 仕様（Swagger 0.33.2 時点）

### 2.1 宛先の追加

- **エンドポイント**: `POST /documents/{documentID}/participants`
- **Content-Type**: `application/x-www-form-urlencoded`
- **主なパラメータ**（抜粋）:
  - `name`（必須）, `email`, `organization`
  - `access_code`, `language_code`, `advanced_authn_request`
  - `tel`, `recipient_id`, `callback`（組込み署名まわり）

**役割（締結者／確認者）や「承認者のみ」を指定する項目はスキーマに存在しない。**

### 2.2 order と「押印者」の関係（重要）

- **`order` は「宛先の順序」だけを表す。** order を設定（＝participants の追加順）しても、**それだけではその宛先が押印者（締結者）になるわけではない。**
- クラウドサインでは**「確認者」と「押印者（締結者）」は別の扱い**である：
  - **締結者（押印者）**: 書類に「押印」「フリーテキスト」「チェックボックス」などの**入力項目が割り当てられている**宛先。その人が入力の上で同意する。
  - **確認者**: その宛先に**入力項目を一切割り当てない**と、画面上は「同意する」ボタンだけが表示され、押印なしで同意のみで完了する＝実質「確認者」として運用できる（ヘルプ上も「入力項目を設定しなくても契約は有効」）。
- つまり「誰が押印者か」は、**participants の order ではなく、「どの participant に入力項目（widgets）を割り当てるか」で決まる。**

### 2.3 participantModel（レスポンス／書類取得時）

- `id`, `email`, `name`, `organization`, `order`, `status`, `access_code`, `language_code`, `processed_at` など。
- **`order`**: 「書類内の宛先の順序（送信者は 0）」。追加順が順番として扱われる。**順番のみ。押印者かどうかは別。**
- **`status`**: 例）0: アクセス不可, 2: 下書き, 3: 配信待ち, 4: 確認待ち, 6: 送信済み, **7: 確認済み**, **8: 押印または入力済み**, 9: 却下, 10: キャンセル, 12: 署名中。結果として「確認だけした人」は 7、「押印または入力した人」は 8 になる。

役割（role）や participant_type のようなフィールドは**定義されていない**。

### 2.4 入力項目（widget）と宛先の紐付け

- **widgetModel**（入力項目）には **`participant_id`** がある：「入力が割り当てられている宛先の ID」。
- エンドポイント: `POST /documents/{documentID}/files/{fileID}/widgets` で押印・フリーテキスト・チェックボックスを追加する際、**どの宛先（participant_id）に割り当てるか**を指定する。
- **ある participant に widget を1つも割り当てない**＝その人は「同意のみ」で完了＝**確認者としての扱い**になる。  
  **ある participant に widget を割り当てる**＝その人は押印または入力の上で同意＝**押印者（締結者）としての扱い**になる。

## 3. APIでできること・できないこと（現仕様ベース）

| 項目 | APIでの可否 |
|------|-------------|
| 宛先を複数追加する | ✅ 可能（複数回 `POST .../participants`） |
| 宛先の順番を決める | ✅ 追加順で決まる（`order` で確認可能） |
| 順番どおりにメールを送る | ✅ 仕様・ヘルプ上、順次送付の想定 |
| 押印者と確認者を**APIで**区別する | ✅ **その宛先に widget を割り当てるかどうか**で表現可能（割り当てない＝確認者） |
| 「承認のみ」の参加者を**APIで**表現する | ✅ 該当 participant に **widget を追加しない** |

## 4. 実装時の進め方

1. **順番付きフローだけ使う場合**  
   - 法務 → 承認者 → 相手 の順に `POST .../participants` で追加し、`POST /documents/{documentID}` で送信。  
   - 全員が「確認・同意」する前提で、API仕様上は全員同じ扱い。

2. **「承認のみ（確認者）」をAPIで表現する場合**  
   - 確認者にしたい宛先には **`POST .../files/{fileID}/widgets` で入力項目（押印・フリーテキスト等）を割り当てない**。  
   - 押印者にしたい宛先の `participant_id` に対してだけ、widget を追加する。  
   - これにより「1人目：確認のみ／2人目：押印／3人目：相手が押印」のようなフローをAPIで組める。

3. **社内承認をクラウドサインの外でやりたい場合**  
   - 社内ツール（ワークフロー・チャット等）で承認完了後に、クラウドサインAPIで書類作成・宛先追加・送信を行う構成が確実。
